# 音素照合マニフェストの用意（`phoneme_alignment_check`）

本書は、[仕様.md §8.1](./仕様.md) の **`text.phoneme_alignment_check`** 用 JSONL を、**別プロセスでどう用意するか**をまとめる。設定キーそのものの説明は仕様書 §8.1 を正とする。

---

## 1. 何のためか

* Common Voice の **`validated.tsv` の `path`（クリップファイル名）** と対応する行ごとに、**音声側から得た音素列**（強制アライメント・別モデル等）を記録する。
* `preprocess` 実行時に、パイプライン内の **G2P（`text.phonemize: true` の OpenJTalk 系）** との **トークン列比較**（編集距離率）で足切りする。
* **マニフェストの音素表記は、G2P のトークンと同一の記号・区切り規則に揃える**こと。表記が違うだけで誤拒否になる（MFA の `japanese_mfa` 等は記号体系が異なることが多いので、そのままでは使えない場合がある）。

---

## 2. ファイル形式（JSON Lines）

1 行 1 JSON オブジェクト。実装は `cv_preprocess/io/alignment_phoneme_manifest.py` を参照。

| フィールド | 必須 | 内容 |
|------------|------|------|
| `source_path` | はい | **`validated.tsv`（または設定の `clip_tsv`）の `path` 列と完全に同一**の文字列（例: `common_voice_ja_12345678.mp3`）。前後空白は読み込み時に除去される。 |
| `phonemes` | はい | **空白区切り**の音素トークン列。G2P 出力と同じ粒度（1 トークン = 比較 1 単位）。 |

重複する `source_path` があれば **後勝ち**。

### 例（2 行）

```json
{"source_path": "clip_abc.mp3", "phonemes": "k o N n i ch i w a"}
{"source_path": "clip_def.mp3", "phonemes": "a i u e o"}
```

実際のトークンは G2P の辞書に依存する。手元の文で確認するには次の CLI が使える。

```bash
cv-preprocess text-normalize "元の文"
cv-preprocess phonemize "正規化後の文"
```

`preprocess` は TSV の文に対し **設定どおりの正規化＋音素化** を行うため、マニフェストを作る側でも **同じ正規化ルール**でテキストを揃えたうえで、比較対象となる「音声由来」の音素列だけを別経路で求める。

---

## 3. 本リポジトリの生成パイプライン（推奨）: `phoneme-manifest`

**`cv-preprocess phoneme-manifest`** が、**`preprocess` と同じ設定**（`input` / `speakers` / `clip_metadata_filters` / `text` のロケール・日本語・長さ・`phonemize` / `g2p_kana`）で TSV を走査し、JSONL を書き出す。実装は `cv_preprocess/pipeline/phoneme_manifest.py`。

### 3.1 モード `g2p_text`（既定）— OpenJTalk G2P に完全一致

* 各行の `sentence` を **`normalize_for_tts`** し、**`pyopenjtalk` の `g2p`**（`cv_preprocess.text.phonemize.g2p_phonemes`）で音素列を得る。
* **`preprocess` がメタデータに書く `phonemes` と同一ルール**のため、`phoneme_alignment_check` で **常に一致**する（照合は実質「設定の回帰テスト」や、後から `text` ルールを変えたかの検知に近い）。
* **用途**: マニフェスト生成・CI・手元での形式確認。音声波形とは独立（テキストのみ）。

```bash
cv-preprocess phoneme-manifest -c config/default.yaml --output data/aligned_phonemes.jsonl
# または config に phoneme_manifest.output_path を書き --output を省略
```

**MFA 足切り（`mfa_gate.compare_phones_to_g2p`）** でも同じ YAML 形式を **`mfa_gate.mfa_to_g2p_token_map_path`** に指定でき、TextGrid 由来の音素列を **変換したうえで** OpenJTalk G2P と比較する（[仕様.md §5.2](./仕様.md)）。

### 3.2 モード `mfa_textgrid` — MFA の phones を YAML マップで OpenJTalk トークン列へ

* 各行の TSV `path` から **`Path(path).stem`** を取り、**`{mfa_textgrid_root}/{stem}.TextGrid`** を読む（Praat long TextGrid。`phones` を名前に含む IntervalTier を `cv_preprocess/audio/textgrid_phones.py` と同じ規則で解釈）。
* 各区間のラベルを時系列に並べた **MFA トークン列**に対し、**YAML 辞書**で **MFA トークン → OpenJTalk G2P 側の空白区切り列**へ変換する（値に複数トークンを含められる。値が空文字のキーはその MFA トークンを出力から省略）。
* **マップに無い MFA トークンはそのまま** JSONL に出る（OpenJTalk と異なる記号のまま残ると照合で不一致になりやすい）。

```bash
cv-preprocess phoneme-manifest -c config/default.yaml \
  --output data/aligned_phonemes.jsonl \
  --source mfa_textgrid \
  --mfa-textgrid-root /path/to/textgrids \
  --mfa-token-map config/mfa_to_openjtalk_phones.yaml
```

### 3.2.1 マップ YAML の草案生成（`suggest-mfa-g2p-map`）

手書きの前に、**TSV の文＋同じ `path` の TextGrid** から (MFA トークン, G2P トークン) の対応を **投票で集計**し、`mfa_to_g2p_token_map_path` / `--mfa-token-map` と同じ形式の **YAML 草案**を出せます（実装: `cv_preprocess/pipeline/mfa_g2p_map_suggest.py`）。

* **長さが一致する行**では MFA と G2P を **1 対 1（zip）** で投票。**一致しない行**では **比例配置**で近似対応を作るため、**誤提案が混ざる**。必ず付随する `*_report.json` の `ambiguous_mfa` / `skipped_low_confidence` を見て人手で直すこと。
* `phoneme-manifest` と同じ **`input` / `speakers` / `text`（`phonemize` 必須）** で TSV を走査する。

```bash
cv-preprocess suggest-mfa-g2p-map -c config/default.yaml \
  --mfa-textgrid-root /path/to/textgrids \
  -o config/mfa_to_openjtalk_phones.generated.yaml
```

既存 YAML を **`--existing-map`** で読み込み、**未登録の MFA キーだけ**埋める（既定）。全体を上書きする場合は **`--overwrite-suggestions`**。

### 3.2.2 NFA（`nfa_gate`）用: `suggest-nfa-g2p-map`

**NeMo Forced Aligner** の token CTM に出るモデル用トークンと OpenJTalk G2P は **既定で同一ではない**（[仕様.md §5.3](./仕様.md)）。`nfa_gate.compare_tokens_to_g2p: true` で音素一致の足切りをするには、**`nfa_gate.nfa_to_g2p_token_map_path`** に、MFA 用と同じ形式の **YAML 辞書**（キー = NFA トークン、値 = G2P 側の空白区切り列）を指定する。

**トークンマップを使わない**代替として、`align_using_pred_text` と **`compare_pred_text_to_norm`** を有効にすると、NeMo の **`pred_text`** と参照 **`text_norm`** を **それぞれ OpenJTalk G2P** した空白区切り音素列で比較する（`text.phonemize: true` が前提。詳細は仕様 §5.3）。

手書きの前に、**`preprocess` と同じ pass1 音声パイプライン**のあとで NFA を走らせ、(NFA トークン, G2P トークン) を **投票で集計**して YAML 草案を出せます（実装: `cv_preprocess/pipeline/nfa_g2p_map_suggest.py`）。

* **長さが一致する行**では NFA 列と G2P を **1 対 1（zip）** で投票。**一致しない行**では **比例配置**で近似対応を作るため、**誤提案が混ざる**。必ず付随する `*_report.json` の `ambiguous_nfa` / `skipped_low_confidence` を見て人手で直すこと。
* `phoneme_alignment_check`（§1）はこのコマンドでは **参照しない**（マニフェスト無しでも集計できる）。

```bash
cv-preprocess suggest-nfa-g2p-map -c config/default.yaml \
  -o config/nfa_to_openjtalk_phones.yaml
```

件数を絞る試行には **`--max-clips N`**。既存 YAML を **`--existing-map`** で読み、**未登録の NFA キーだけ**埋める（既定）。全体を上書きする場合は **`--overwrite-suggestions`**。

### 3.3 設定ファイル（任意）`phoneme_manifest:`

`config/default.yaml` 等にブロックを置くと、CLI の `--output` / `--source` 等を省略できる。

```yaml
phoneme_manifest:
  output_path: data/aligned_phonemes.jsonl
  source: g2p_text
  # mfa_textgrid のとき:
  # mfa_textgrid_root: out/mfa_textgrids
  # mfa_token_map_path: config/mfa_to_openjtalk_phones.yaml
```

### 3.4 レポート

出力 JSONL と同じディレクトリに **`{stem(出力ファイル名)}_report.json`**（例: `aligned_phonemes_report.json`）が付く。件数・スキップ理由・警告を含む。

## 4. 用意の流れ（手動・他ツールと組み合わせる場合）

1. **母集団の決定**  
   `preprocess` が読む **`clip_tsv` の `path` 列**について、照合したいクリップだけマニフェストに載せる（全件でも可）。
2. **G2P 側の基準の確認**  
   対象文を `text-normalize` → `phonemize` し、トークン列の見え方を固定する。
3. **音声由来音素列の生成（外部ツール）**  
   強制アライメント・ASR の音素出力・自前モデルなど **任意**。ここで得た記号を、必要なら **G2P と同じ語彙に正規化**（置換テーブル、ルール、別の辞書で再音素化など）。**MFA → OpenJTalk への変換**は §3.2 のマップ付き `phoneme-manifest` で自動化できる。
4. **JSONL の書き出し**  
   各行 `{"source_path": "<TSV の path>", "phonemes": "..."}`。UTF-8。
5. **設定**  
   `text.phoneme_alignment_check.enabled: true` と `manifest_path`（ファイルが存在すること。無いと起動時に失敗する）。  
   `missing_manifest_entry`: マニフェストに無い `path` は `skip`（比較しない）か `reject`（拒否）かを選ぶ。

## 5. 手段の例（パイプラインの外で行う）

### 5.1 自前スクリプトで TSV を走査して生成する

* `validated.tsv` を読み、`path` と `sentence`（および必要なら `client_id`）を取り出す。
* `sentence` を **本パイプラインと同じ正規化**に通す（`cv_preprocess.text.normalize.normalize_for_tts` を import するか、CLI で確認した結果と一致する処理を書く）。
* 音声ファイルのパスを `corpus_root` / `audio_subdir` / `path` で組み立て、**別ジョブ**でアライメントや推論を実行する。
* 得た音素列を G2P 互換に変換したら、`source_path` は **TSV の `path` の生文字列**をそのまま使う（`clips/` を付けない等、TSV と一致させる）。

### 5.2 Montreal Forced Aligner（MFA）の結果を使う場合

* `mfa align` 等で **TextGrid** を得る。
* 区間ごとの音素ラベルを時系列に連結し、**1 発話 1 行のトークン列**にする。
* **重要**: MFA の音素記号は OpenJTalk G2P と一致しないことが多い。**§3.2** の **`phoneme-manifest --source mfa_textgrid` + トークンマップ YAML** で OpenJTalk 列に寄せるか、自前で同等の変換を行う。  
  本リポジトリの **`mfa_gate.compare_phones_to_g2p`** は別用途（バッチ MFA 足切り）であり、**`phoneme_alignment_check` のマニフェスト形式とは独立**である。

### 5.3 piper_plus 等の下流で既に音素タイミングがある場合

* 学習パイプライン側で **同じ `path`（または一意に対応付け可能な ID）** と紐づく音素列をエクスポートし、本書の JSONL 形式に変換する。
* 記号は再び **G2P と同一**であることを確認する。

### 5.4 検証用に空や最小ファイルから始める

* 開発時は **数行だけ**の JSONL を置き、`max_token_error_rate` を緩めて分布を見る。
* 本番で厳密一致のみ通すなら `max_token_error_rate: 0` と仕様書どおり。

---

## 6. パス・カレントディレクトリの注意

* `manifest_path` は **YAML 上の相対パスは、コマンド実行時のカレントディレクトリ基準**で解決される（通常はリポジトリルートで `cv-preprocess preprocess -c config/default.yaml` を実行する想定）。
* 迷う場合は **設定ファイルからの相対ではなく、絶対パス**を書くか、常に同じ cwd でジョブを回す運用にする。

## 8. 設定で有効化する例（抜粋）

```yaml
text:
  phonemize: true
  phoneme_alignment_check:
    enabled: true
    manifest_path: data/aligned_phonemes.jsonl
    missing_manifest_entry: skip
    max_token_error_rate: 0.0
```

`enabled: true` にする前に **`manifest_path` のファイルが存在する**ことを確認すること。まず **`cv-preprocess phoneme-manifest`** で生成する運用を推奨する。
