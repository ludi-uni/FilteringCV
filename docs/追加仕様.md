# 多話者データセット構築のための前処理（追加仕様）

## 0. 位置づけ

本ドキュメントは、[仕様.md](./仕様.md) の **Common Voice 前処理パイプライン**を前提に、**piper_plus 等の別リポジトリで多話者 TTS を学習する**ために必要な **データセット構築工程（前処理の拡張・出力の揃え方）**だけを定義する。

| 段階 | 責務 |
|------|------|
| CV 読み込み・話者／**属性フィルタ**・音声チェーン・品質ゲート・JSONL 等 | [仕様.md](./仕様.md) |
| **多話者コーパスとしての** QC 方針（足切り）、話者バランス・エクスポート形式 | **本書** |
| 強制アライメントの実行・学習 | **別ツール・piper_plus**。本リポジトリは **§6 品質ゲート**に加え、**任意で音素照合足切り**（[仕様.md §8.1](./仕様.md)） |

---

## 1. 全体フロー（本リポジトリ側の論理）

```text
CV raw（公式アーカイブ）
  ↓
メタデータで絞り込み（client_id + 任意で gender / age 等 §2.4）
  ↓
G2P と外部マニフェストの音素照合（任意、§8.1）→ ずれ大なら除外
  ↓
音声前処理（**足切り前**のノイズ除去は Dasheng — 下記 §2）
  ↓
**MFA**（任意、[仕様.md §5.2](./仕様.md)）— アライメント失敗や音素不一致で除外
  ↓
品質ゲート（§6）で足切り
  ↓
多話者向けメタデータ・WAV のエクスポート
  ↓
（以降は piper_plus 等でアライメント・学習）
```

* （任意）**二次パイプライン**（**§11**）— 一次で採用した WAV に補正（**HiFi-GAN 帯域補完**は **§12**）を適用し、**再品質ゲート**でティア振り分けしてから export する場合は、一次 export のあと別ジョブで実行する。

---

## 2. ノイズ除去（品質ゲート前）

一次の `denoise`（**品質ゲートより前**）では **[mispeech/dasheng-denoiser](https://huggingface.co/mispeech/dasheng-denoiser)** を想定（[仕様.md §5](./仕様.md)、`method: dasheng`）。依存は `uv sync --extra dasheng`（[開発環境.md](./開発環境.md)）。

---

## 3. クリップ属性による入力絞り込み

[仕様.md §2.4](./仕様.md) の `speakers.clip_metadata_filters` で性別・年齢・アクセント等を絞る（実装済み）。

---

## 4. 足切り（QC）の目安

採否の本体は [仕様.md §6](./仕様.md) の `quality_gate`。任意で **[仕様.md §8.1](./仕様.md) `phoneme_alignment_check`**（手順は [音素照合マニフェスト.md](./音素照合マニフェスト.md)）、**[§5.2](./仕様.md) MFA**、**[§5.3](./仕様.md) NFA** を併用できる。MFA / NeMo の導入は [開発環境.md](./開発環境.md)。閾値は多話者向けに設定プロファイルで分けてよい。

| 項目 | 例 |
|------|-----|
| SNR | `min_estimated_snr_db` 等 |
| 長さ | `min_duration_sec` / `max_duration_sec` |
| テキストと音声 | モーラ下限・無音率等（仕様.md §6） |

---

## 5. 多話者データセットの構造とメタデータ

piper_plus が期待する形式に合わせてエクスポートする。**最終的なファイル名・列名は piper_plus 側の仕様に合わせてマッピング**する（本書は論理レイアウトを示す）。

```text
dataset/
 ├── wavs/
 ├── metadata.csv    # または metadata.jsonl（[仕様.md §13](./仕様.md)）
 ├── speakers.json   # 任意: 話者メタ・ID 対応表
```

### 5.1 必須の論理列（多話者）

| 論理名 | 内容 |
|--------|------|
| 音声パス | 前処理済み WAV への参照 |
| テキスト | 正規化テキスト（学習先と同一ルール） |
| speaker_id | Common Voice の `client_id` 等 |

音素・duration は **piper_plus 側でアライメントして付与**する想定。本パイプライン出力に含める場合は [仕様.md §8・§13](./仕様.md) と学習先の契約を合わせる。

### 5.2 話者バランス（データセット構築時）

* **話者あたり上限クリップ数**（例: 最大 2000 / speaker）でキャップする。
* または **重み付け用のメタデータ**を `speakers.json` 等に書き、下流で利用可能にする。

**注意**: 全話者を無理に均等化しすぎると総量が減る。キャップと除外理由は `report.json` に残す。

---

## 6. train / val / test（多話者）

[仕様.md §10](./仕様.md) に従い、**speaker-aware** で同一 `client_id` が split を跨がないようにする。

---

## 7. 任意のデータ拡張（前処理段階）

* pitch 補正、speed perturbation（±5% 程度）など。
* loudness normalization（例: **-23 LUFS**、[仕様.md](./仕様.md) の `normalize.method: loudness` と整合）。

---

## 8. 失敗パターン（データセット構築時）

| 避けるべきこと | 理由 |
|----------------|------|
| 全話者を機械的に均等化しすぎる | データが減り、多話者の多様性が損なわれる |
| ノイズ除去のやりすぎ | 音色が損なわれ、下流学習が不安定になりうる |
| `clip_metadata_filters` と空欄メタデータの組み合わせ | 属性未提出の行がすべて落ち、件数が 0 になりうる |
| speaker_id の欠落・衝突 | 多話者ラベルが壊れる |

---

## 9. まとめ

* 本リポジトリのゴールは **フィルタ・Dasheng 前処理・品質ゲート後の WAV + メタデータのエクスポート**まで（一次パイプライン）。
* **二次パイプライン**（補正 → 再品質振り分け）と **HiFi-GAN 帯域補完**は **§10–§12** の計画に従い、実装・設定を拡張する。
* **アライメント・学習は piper_plus**（別リポジトリ）に委ねる。

---

## 10. Common Voice 品質足切りパイプライン（計画）

**目的**: Mozilla Common Voice 由来のクリップを、**再現可能な段階**と **拒否理由のトレーサビリティ**を保ったまま、TTS 学習に使う母集団へ絞り込む。既存の [仕様.md §4](./仕様.md)（全体フロー）・[§6](./仕様.md)（品質ゲート）を **論理パイプライン**として明示する。

### 10.1 段階モデル（一次・論理）

| 段 | 名称 | 内容 | 主な参照 |
|----|------|------|-----------|
| G0 | 入口（メタ・テキスト） | `validated.tsv` 起点、`client_id` / `clip_metadata_filters`、ロケール・日本語判定、テキスト正規化 | [仕様.md §2・§7](./仕様.md) |
| G1 | 音素・外部照合（任意） | G2P と外部マニフェストの音素照合で大きくずれるクリップを除外 | [仕様.md §8.1](./仕様.md) |
| P | 音声改善チェーン | デコード、エッジ UI クリック対策、Dasheng 等の denoise、normalize、trim 等（順序は設定） | [仕様.md §5](./仕様.md)、本書 §2 |
| G2 | アコースティック整合（任意） | MFA による強制アライメント失敗・（設定時）音素不一致で除外 | [仕様.md §5.2](./仕様.md) |
| G3 | **品質ゲート（一次）** | 長さ、残存無音、クリッピング、推定 SNR、テキスト長との整合など | [仕様.md §6](./仕様.md) |
| Out | 採用・記録 | 通過分はメタデータ行、落ちた分は `rejects` と理由コード、`report.json` に集計 | [仕様.md §13](./仕様.md) |

**方針**: 「生波形が悪いから即除外」ではなく、**P のあと**に G3 を掛ける（[仕様.md §3](./仕様.md)）。

### 10.2 閾値・プロファイル（運用）

* 多話者では録音環境差で SNR 等のばらつきが大きい。**グローバル閾値**に加え、本書 §5.2 の **話者あたりキャップ**や、設定での **`quality_gate` プロファイル**（例: `strict` / `balanced` / `lenient`）を分けられるようにするのが望ましい（実装時に YAML キー名を確定する）。
* 閾値は **`preprocess` 後の `report` に載る指標分布**を見てキャリブレーションする。初回は緩めに通し、ヒストグラムで切り口を決める。

### 10.3 実装マイルストーン（案）

1. **M1**: G0→P→G3 までの拒否理由・指標を **クリップ単位で完全にログ**（既存機能の確認と不足フィールドの補完）。
2. **M2**: `quality_gate` の **複数プロファイル**を設定で切り替え、`report` にプロファイル名を記録。
3. **M3**: G1・G2 を有効にしたときの **拒否率サマリ**（段階別件数）を `report` に固定フォーマットで出力。

## 11. 補正後の再品質振り分けパイプライン（二次パイプライン）（計画）

**目的**: **一次品質ゲート（§10 の G3）を通過した波形**（または、運用上「境界帯」を別キューに逃がしたクリップ）に対し、**追加の信号処理・生成モデル**で改善し、**同じまたは拡張した品質指標で再評価**して、学習への投入ティアへ振り分ける。

### 11.1 一次との関係

```text
一次: CV → … → 品質ゲート（G3）→ 採用 WAV / メタデータ
                    ↓
二次: 採用（または境界帯）WAV → 補正チェーン → 再スコア → ティア A/B/C（等）→ export
```

* **入力**: 一次出力の WAV パス、一次時点の `quality` 系メタデータ（**補正前指標として保持**）。
* **出力**: 補正後 WAV（または上書きポリシーを設定で選択）、`quality_tier`（論理名）、**二次指標**、適用したステップ列（再現性用）。

### 11.2 補正ステップ（候補）

実装は一次と同様 **宣言的チェーン**がよい。例:

| 候補 | 役割 | 注意 |
|------|------|------|
| `normalize`（再） | 一次後にブレたラウドネスをターゲットへ寄せる | 過剰ゲインでクリッピング再発を防ぐ |
| `denoise`（第2パス） | パラメータ弱め・帯域制限付き等、一次とは別プロファイル | 音色劣化。必ず二次ゲートで確認 |
| **`bandwidth_extension`（HiFi-GAN）** | 高域の帯域補完 | **§12**。アーティファクトに敏感なため再ゲート必須 |
| （将来）`dereverb` 等 | 残響抑制 | 利用する場合はライセンス・学習ドメインを別途整理 |

**禁止に近い方針**: 発話内容や話者性を変えるような **大きなピッチシフト・タイムストレッチ**は、TTS コーパス用途では原則載せない（研究用フラグで切り出すなら可）。

### 11.3 再品質ゲートと振り分け

* **再ゲート**: 補正後に **G3 と同等の指標**を再計算する。HiFi-GAN 等を使う場合は **スペクトル急峻度・高域帯のノイズフロア・（任意）知覚スコア**など、**生成由来のアーティファクト**を捉える指標の追加を検討する。
* **ティア例**:
  * **A**: 現行 §6 と同水準（そのまま主学習に投入）。
  * **B**: 一部閾値を緩和するが、別条件（例: 残存ノイズ上限）で抑える。**重み下げやサブセット学習**用。
  * **C**: 学習から除外、または研究・デバッグ用に隔離。

### 11.4 データ分割との順序

* **train / val / test**（[仕様.md §10](./仕様.md)）は、**最終ティア（二次の振り分け）が確定した母集団**に対して行う。境界帯を補正で「昇格」させる場合も、**シード・手順を `report` に残し**再現可能にする。

### 11.5 実装マイルストーン（案）

1. **S1**: 設定に `secondary_audio_chain`（仮名）を追加し、一次出力ディレクトリを入力に取る **サブコマンドまたは `--stage secondary`** を定義。
2. **S2**: メタデータスキーマに `primary_quality` / `secondary_quality` / `corrections_applied` / `quality_tier` を追加（[仕様.md §13](./仕様.md) と整合）。
3. **S3**: `export` が **ティア列**を下流（piper_plus 等）の契約にマッピング可能にする。

## 12. HiFi-GAN による帯域補完（追加仕様）

**目的**: ローパス・電話帯域・低ビットレート由来などで **高域が欠けた音声**を、目標サンプリングレート・帯域まで **ニューラルボコーダ（HiFi-GAN 系）**で補い、TTS 前処理済みコーパスとしての品質を揃える。**学習用の Universal HiFi-GAN** や、**日本語・読み上げドメインにファインチューンした**チェックポイントの利用を想定する。

### 12.1 パイプライン上の位置

* **推奨**: **二次パイプライン（§11）内**で適用し、**補完直後に再品質ゲート**を通す。一次ゲートの**前**だけに帯域補完を置くと、ゲート指標が「生成後の分布」に引きずられ、一次閾値設計が曖昧になりやすい。
* **例外**: 実験的に「生に近い 8 kHz → 24 kHz 拡張のみ」を先に行い、その後に denoise するなど、チェーン順は **設定で明示**し `report` に記録する。

### 12.2 入出力・前提

* **入力**: 単一チャンネル PCM、**実効帯域**または**意図したローカット周波数**をメタデータまたは設定で明示（一次チェーンの `lowpass` と整合）。
* **出力**: ターゲットレート（例: 24000 Hz または 48000 Hz）の WAV。以降のステップ（normalize 等）は **出力レート基準**で定義する。

### 12.3 リスクと緩和

| リスク | 緩和 |
|--------|------|
| メタリックな高域・ブレス増幅 | 再ゲートでスペクトル・クリップ・SNR を確認；モデル・推論設定の A/B |
| 学習データと CV のドメイン差 | ファインチューン、または「帯域補完対象外」ティアに分離 |
| 推論コスト | GPU バッチ化、二次は一次より件数が少ない前提でのジョブ設計 |

### 12.4 依存・ライセンス

* 実装は [HiFi-GAN（例: jik876/hifi-gan）](https://github.com/jik876/hifi-gan) 等を参照し、**ライセンス・学習データ条項**に従う。重みファイルの再配布可否は利用モデルごとに確認する。
* 依存追加は [開発環境.md](./開発環境.md) に **オプション extra**（例: `hifigan`）として追記する想定。

### 12.5 設定キー（案・実装時に確定）

```yaml
# 二次チェーン内の例
bandwidth_extension:
  enabled: true
  backend: hifigan
  checkpoint: null  # パス、または Hugging Face モデル ID（実装方針に合わせる）
  input_sample_rate: 24000
  output_sample_rate: 24000
  # 一次で適用したローカットと整合させる場合
  assumed_bandwidth_hz: 8000
```

### 12.6 実装マイルストーン（案）

1. **H1**: 推論のみ（固定チェックポイント）を CLI から呼べるラッパー。
2. **H2**: `bandwidth_extension` ステップを §11 のチェーンに組み込み、**再ゲート結果を `report` に出す**。
3. **H3**: （任意）CV 日本語サブセットでの **軽量ファインチューン手順**を別ドキュメントまたはノートブックで整備。
