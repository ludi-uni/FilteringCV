## 1. 目的

Common Voice から **TTS 学習に使える日本語音声コーパス**を生成する。収録品質の都合で **ノイズが乗りやすく、前後無音や区切り無音が長いクリップが多い**前提とし、**まず除外ではなく音声前処理で改善**し、そのうえで **受入基準を満たした発話だけ**を学習用に登録する。

最終的に次を出力する。

* 前処理済み音声（WAV 等、設定で指定した処理チェーンの結果）
* 元テキストおよび正規化テキスト（および任意で音素列）
* 品質・由来・適用した前処理のメタデータ
* 本パイプラインで定義する **train / val / test** の分割（以下 **val** は検証用。フレームワークが `dev` 名を期待する場合は、設定で **val と dev のどちらのファイル名・列値で書き出すか**を揃える）

## 2. 入力データのディレクトリ構造（Common Voice 側）

公式の言語パック（`.tar.gz` 展開後）は、概ね次の階層になる（`{lang}` は BCP 47、日本語は `ja`）。

```text
cv-corpus-{version}-{YYYY-MM-DD}-{lang}.tar.gz/
  cv-corpus-{version}-{YYYY-MM-DD}/
    └── {lang}/
        ├── README.md                 （Corpus 25.0 以降: データシート）
        ├── clips/
        │   └── *.mp3
        ├── validated.tsv
        ├── invalidated.tsv
        ├── other.tsv
        ├── train.tsv
        ├── dev.tsv
        ├── test.tsv
        ├── reported.tsv
        ├── clip_durations.tsv        （Corpus 16.1 以降）
        ├── validated_sentences.tsv
        └── unvalidated_sentences.tsv
```

### 2.1 本パイプラインで「正」とする入力

* **音声**: `{lang}/clips/` 以下のファイル（TSV の `path` 列と対応）。実装上の絶対パスは **`corpus_root` / `audio_subdir`（既定 `clips`）/ `path`** とする。
* **クリップ一覧**: 原則 **`validated.tsv`** を第一ソースとする（TTS では検証済みクリップが起点になりやすい）。
  * `train.tsv` / `dev.tsv` / `test.tsv` は CorporaCreator による**公式分割**であり、行集合は `validated.tsv` の一部で、かつ**リリースごとに非決定的に変わり得る**。本仕様では **TTS 用に自前で train/val/test する**ため、これらは参照・デバッグ用とし、必須入力にはしない。
* **補助**: `clip_durations.tsv`（ミリ秒）を使えば、全件デコードせずに粗い長さの目安を付けられる（実装は任意）。

### 2.2 クリップ TSV の列（Scripted Speech）

`validated.tsv` / `invalidated.tsv` / `other.tsv` および `train` / `dev` / `test` の各ファイルは、**行あたり1クリップ**で、列は共通である。

| 列名 | 説明 |
|------|------|
| `client_id` | 話者に相当するハッシュ ID |
| `path` | `clips/` からの相対パス（ファイル名） |
| `sentence` | 読み上げテキスト |
| `sentence_id` | 文 ID（Corpus 17.0 以降） |
| `sentence_domain` | 文のドメイン分類（17.0 以降） |
| `up_votes` / `down_votes` | 検証投票 |
| `age` / `gender` | 提供時のみ |
| `accents` | アクセント（**Corpus 17.0 以降は `accent` ではなく `accents`**） |
| `variant` | 言語バリアント（13.0 以降） |
| `locale` | ロケール（5.0 以降） |
| `segment` | カスタムセグメント（5.0 以降） |

**検証カテゴリ**（公式定義）: `validated` / `invalidated` / `other` の意味は cv-dataset の「Validation Categories」に従う。

**レガシー注意**: 旧リリースや独自コピーでは列名が省略・旧名の場合がある。実装は **「release 差を吸収するパーサ」** を用意し、必須列の欠落はエラーまたは明示的なフォールバックとする。

**実装メモ（必須列の目安）**: 最低限 **`path` と `sentence` と `client_id`** は TTS パイプラインの前提とし、欠ける行は読み込み時にスキップまたはエラーとするかを設定で選べるようにするとよい。

### 2.3 本リポジトリ側の想定配置

ダウンロードしたコーパスは **リポジトリ外**でもよいが、設定では次のような **「言語ルート」** を1パスで指せるようにする。

```text
（例）プロジェクト外
  D:/datasets/cv-corpus-25.0-2026-03-09/ja/

（例）プロジェクト内（gitignore 推奨）
  FilteringCV/data/cv-corpus-25.0-2026-03-09/ja/
```

### 2.4 クリップ属性フィルタ（任意）

TSV に含まれる **話者メタデータ**（提供されている場合のみ）で、**デコード前**に処理対象行を絞り込める。列は **§2.2** のとおり（例: `gender`, `age`, `accents`, `variant`, `locale`, `segment`）。

* 設定キー: `speakers.clip_metadata_filters`（各軸は**許容値のリスト**。**空リスト＝その軸では絞り込まない**）。
* **マッチング**: 値は前後空白除去のうえ **大文字小文字を区別しない**（`casefold` 相当）。TSV セルが**空**の行は、その軸に許容リストを指定しているとき、原則 **不一致扱いで除外**される。**許容リストに要素として明示的に空文字列 `""` を含めた場合のみ**、その軸について空セルの行も許容する（例: `accents: ["標準語", ""]`）。
* **複数軸**を指定した場合は **AND**（すべての指定軸で許容値に含まれる行だけ残す）。
* `input.locale_expected`（§4 手順 3 のロケールチェック）とは別。`clip_metadata_filters.locales` を使う場合は **行の `locale` 列**が許容リストに含まれることも要求する。

`scan` / `preprocess` の `report.json` に、話者フィルタ後件数・属性フィルタ後件数・適用した `clip_metadata_filters` を記録する。

## 3. 設計方針：除外より前処理を先に

* Common Voice のクリップは **ノイズ混じり・無音区間が長い**ことが多いが、方針として **生のまま「品質不足」で捨てる前に、設定に従った音声前処理を適用**する。
* 収録 UI により **録音開始・終了のキー操作音が短いトランジェント（クリック）として乗る**ことがある。**クリックのあと無音が続いて発話が始まる**ケースと、**発話・無音のあとクリックで終わる**ケースを想定し、**5.1 録音 UI 由来のキーボードクリック**の **エッジ処理**で可能な限り除去する。
* **話者（`client_id`）は config で指定**し、対象話者に属するクリップだけをデコード・WAV 化・処理チェーンに載せる（全コーパス一括はオプション）。**性別・年齢などのクリップ属性**でさらに絞り込む場合は **§2.4** の `clip_metadata_filters` を使う。
* 前処理後にのみ **受入基準**（長さ、残存無音率、クリッピング、推定 SNR など）を評価し、**基準を満たした発話に限り**メタデータ行を生成する。
* 基準を満たさないものは **`rejects.csv`（または設定でパス指定）に理由付きで記録**する。可能なら中間ファイルを残すかは設定で選べる。

## 4. 全体フロー（処理順は固定）

以下の番号は実装・ログ・エラー参照で共通に使う（「手順 5」＝受入基準など）。

1. **メタデータ読み込み**（`validated.tsv` 起点）
2. **話者フィルタ** — `config` に列挙した `client_id` の行だけ残す（**`include_client_ids` が空のときは「全行を対象」**とする。別モードが必要なら設定キーで明示する）
2b. **クリップ属性フィルタ**（任意）— **`speakers.clip_metadata_filters`** で §2.4 に従い行を絞り込む（未設定または全軸空なら何もしない）
3. **テキスト側** — ロケール・日本語判定、文字正規化、（任意）音素化
3b. **音素照合**（任意）— **`text.phoneme_alignment_check`** で G2P と外部マニフェストのアライメント音素を比較し、不一致ならこの時点で拒否（§8.1）
4. **音声前処理**（対象クリップごと、順序は `config` のチェーン定義に従う）
   * 例: デコード → リサンプル → **エッジのキーボードクリック対策** → ノイズ抑制 → ノーマライズ → 無音トリム → ローパス → その他
4b. **強制アライメント足切り**（任意）— **§4.1** のとおり、`two_pass_denoise` が無効なら **ノイズ除去など手順 4 チェーン直後**の波形に対し、有効で **完全分離でない**なら **第 1 パス（denoise を除く単一チェーン）直後**、**完全分離**なら **`audio_pipeline_align` 完了直後**の波形に対し、**`mfa_gate` または `nfa_gate` のどちらか一方**（同時有効不可）でバッチ実行する。**MFA**: [Montreal Forced Aligner](https://montreal-forced-aligner.readthedocs.io/)（`mfa align`）、拒否理由は **§5.2**。**NFA**: [NeMo Forced Aligner](https://docs.nvidia.com/nemo-framework/user-guide/latest/nemotoolkit/tools/nemo_forced_aligner.html)（`align.py`）、拒否理由は **§5.3**。
4c. **早期音声ゲート**（任意）— **`early_audio_gate`** で、デコード直後に **ターゲット SR のみ**合わせた波形に対し、`quality_gate` の閾値のうち **有効化した指標だけ**を検査し、落ちたクリップは **重いチェーン（Dasheng 等）を実行せず**拒否する（理由コードは `early_gate_*`）。
4d. **二段 denoise**（任意）— **`two_pass_denoise.enabled: true`** のとき、手順 4 は **第 1 パス用の音声チェーン**で実行し、手順 4b（**MFA または NFA 足切り**）を経た **採用候補に限り**、**第 2 パス**でノイズ除去等を適用してから手順 5 の品質ゲートと WAV 出力に進む。失敗時は `two_pass_denoise_failed:*`。**完全分離**（`audio_pipeline_align` と `audio_pipeline_enhance` の両指定）のときは第 1 パス＝`align` のみ・第 2 パス＝`enhance` の `run_steps_on_array` 全文。従来は単一 `audio_pipeline` に対し第 1 パス＝全 `denoise` を `none` 相当にしたチェーン、第 2 パス＝元の非 `none` の `denoise` のみ再適用。
5. **受入基準（品質ゲート）** — 前処理後の波形に対して閾値判定
6. **重複・話者偏りの調整**（任意、メタデータ生成前または分割前）
7. **メタデータ生成** — 通過した発話のみ `metadata` に出力
8. **train / val / test 分割** — **speaker-aware** を推奨（同一 `client_id` が val/test と train に跨がない）
9. **エクスポート** — 最終配置・マニフェスト・`report.json`

### 4.1 早期ゲート・二段 denoise・MFA 並列（任意）

* **`early_audio_gate`**（ルートキー）— `enabled` と各 `check_*`（`check_duration` / `check_silence_ratio` / `check_snr` / `check_clipping` / `check_dc_offset` / `check_chars_per_sec` / `check_mora_duration`）。閾値は **`quality_gate`** と **`snr`** をそのまま参照（未処理波形では SNR・無音は厳しすぎることがあるため、既定は duration・クリップ・DC のみ）。
* **`two_pass_denoise`** — `enabled: true` で **§4 手順 4** を第 1 パス用チェーンで実行し、**§4 手順 4b** の **MFA または NFA** はその波形に対して実行。**通過後**に第 2 パスで denoise 等を適用。GPU 負荷の高い denoise を足切り後に限定する用途。**`audio_pipeline_align` / `audio_pipeline_enhance`** を両方指定すると単一 `audio_pipeline` の「denoise スキップ」ではなく **足切り用・加工用を完全に別定義**する（`target_sample_rate` は一致必須）。`report.json` に `two_pass_split_pipelines`・各 `audio_pipeline_*_id` が付く。**`enhance_phase`** — `interleaved`（既定）で MFA/NFA バッチ通過のたびに第 2 パスへ進む。`after_align_complete` は **全行の align フラッシュが終わった後**にだけ第 2 パスをまとめて実行（NFA 常駐子を閉じた後に SGMSE を開始しやすい。**ホスト RAM** は通過クリップ分だけキューが増える）。
* **実装上の最適化（`two_pass_denoise`）** — **完全分離でない**とき、非 `none` の `denoise` が **SGMSE のみ**のとき、MFA/NFA の **バッチ通過分**を最大 **`two_pass_denoise.sgmse_micro_batch_max`** 本（既定 **8**）ずつまとめて `enhance_batch` に載せる。完全分離時は **第 2 パスが SGMSE のみ**のとき同様（`audio_pipeline_enhance` のステップで判定）。失敗時は当該チャンクをクリップ単位の従来処理にフォールバック。`metadata.jsonl` の `two_pass_denoise` に `sgmse_micro_batch` や `mode: split_enhance` が付くことがある。
* **`mfa_gate.auto_num_jobs`** — `true` のとき `mfa align` の `--num_jobs` を **`max(1, os.cpu_count())`** に合わせる（`report.json` に `num_jobs_resolved` を記録）。
* **`mfa_gate.auto_scale_batch_size`** — `true` のとき `batch_size = max(設定値, min(batch_size_max, num_jobs_resolved * auto_batch_jobs_multiplier))`（`report.json` に `batch_size_resolved`）。
* **`mfa_gate.prefilter`** / **`nfa_gate.prefilter`** — **§5.2** / **§5.3** のとおり、各アライナのバッチ投入前に品質ゲートで母集団を減らす（`mfa_prefilter_*` / `nfa_prefilter_*`）。

## 5. 音声前処理（config で段階・パラメータを指定）

実装は **プラグイン的なステップ列**とし、有効なステップと順序を YAML で宣言する。以下は例（アルゴリズム名は実装時に固定）。

| ステップ例 | 役割 |
|------------|------|
| `decode` | MP3 等から PCM へ（設定に明示しない場合も **パイプライン入口で必ず実行**する想定でよい） |
| `resample` | 目標サンプルレート・位相 |
| `denoise` | ノイズ抑制。**品質ゲート（§6）より前**に適用する `denoise` では、既定ポリシーとして **[mispeech/dasheng-denoiser](https://huggingface.co/mispeech/dasheng-denoiser)** を使う（`method: dasheng`、`dasheng_model_id` 既定値 `mispeech/dasheng-denoiser`）。その他（`spectral_subtract` / `sgmse` / **`wpe_deepfilternet`**（NARA-WPE＋[DeepFilterNet](https://github.com/Rikorose/DeepFilterNet) 公式学習済み・MIT/Apache-2.0 デュアル）/ `none` 等）は設定で明示。`wpe_deepfilternet` は ``uv sync --extra wpe_dfn`` |
| `normalize` | ラウドネス / ピーク正規化など（**音声**用。テキスト正規化と混同しない） |
| `trim_silence` | 先頭末尾または VAD ベースの無音除去 |
| `edge_ui_click` | 録音開始／終了付近のキーボードクリック等の短いトランジェント検出・除去（**5.1**） |
| `lowpass` / `highpass` | ローパス・ハイパス（Anti-aliasing も含め設計） |
| `save_wav` | 中間または最終 WAV の書き出し |

* **順序は設定どおり**（例: ローパスを無音カットの前後どちらにするかは実験で変わるため、設定で明示）。
* 各ステップの **入出力サンプルレート・チャンネル数**をログまたは `report` に残す。
* 処理に失敗したクリップは **reject**（理由: `decode_failed`, `nan_inf_audio` 等）。理由コードは列挙型またはドキュメント化した文字列とし、**13.1** の rejects と整合させる。

### 5.1 録音 UI 由来のキーボードクリック（エッジ・トランジェント）

録音の開始／終了操作に伴い、**クリックに続いて無音が続き発話が始まる**パターンと、**発話・無音のあとにクリックで終わる**パターンが混在する。単純な無音トリムだけでは前者のクリックが残ったり、後者で発話末尾を誤って切ったりするため、**短い広帯域的なトランジェント**として検出し、**先頭は除去してから無音トリム**、**末尾はクリック手前で切る**方針とする。

#### 想定パターン

| パターン | 時間軸のイメージ | 望ましい処理 |
|----------|------------------|--------------|
| **A: 先頭クリック → 無音 → 発話** | `[click][silence][speech ...]` | クリック区間をミュート／窓かけ除去したうえで、先頭無音を `trim_silence` で削る |
| **B: 発話 → 無音 → 末尾クリック** | `[... speech][silence][click]` | **クリックより手前**で終端を決める（発話本体は残す） |

#### 検出の目安（実装はパラメータ化）

キーボード・UI クリックは次の特徴を持ちやすい。**発話の破裂音・歯擦音と区別**するため、**絶対時間位置**（先頭・末尾の数 ms〜数十 ms に限定）と **短時間幅**の両方を使う。

* **短時間**: 典型として数 ms〜数十 ms（上限は `max_transient_ms` 等で設定。長すぎるイベントは発話成分として扱わない）
* **エネルギー**: 直前直後の低エネルギー区間（無音に近い）に対して、ピークが著しく大きい
* **立ち上がり**: 振幅・包絡の **急峻な立ち上がり／立ち下がり**（サンプル差分や Teager エネルギー等で補助）
* **スペクトル**: 広帯域に近い成分が混ざることがあるが、言語・マイクで変わるため **閾値は相対的**（周囲ノイズフロアに対する比）にする

#### アルゴリズム要件（論理）

1. **先頭（パターン A）**
   * 波形先頭からスキャンし、**クリップ先頭から `lead_scan_ms` 以内**にのみ「クリック候補」を探す（発話中盤の破裂音を誤検出しない）。
   * 候補区間は **最大 `max_transient_ms`** でキャップし、当該区間を **ゼロ埋め／短いフェードアウト／スペクトル寄りの置換**のいずれかで除去（実装選択可。設定で統一）。
   * 続く **無音区間**は既存の `trim_silence` と同じ閾値で切り、**発話開始位置**に揃える。
2. **末尾（パターン B）**
   * 末尾から逆向きにスキャンし、**末尾 `trail_scan_ms` 以内**のみクリック候補を探す。
   * **無音または低エネルギー区間の直後／直前**にある短いスパイクを、末尾クリックとして扱う（「しゃ」「す」などの語尾との区別のため **時間幅上限**と **無音との隣接**を必須条件にする）。
   * 確定したら **クリック開始サンプルより前**を最終出力の終端とする（＝クリック部分を捨てる）。無音が長い場合は **発話終了＋許容トレール無音**まで残すかは `config` で選択可能にする。
3. **安全側のデフォルト**
   * 検出に自信がない場合は **除去しない**（`report` に `edge_click_skipped: uncertain` 等）。誤って発話を削るよりクリック残りを選ぶ。
   * **先頭**: 設定 `lead_click_requires_pre_silence_ms`（任意）を与えると、**RMS ピークより左に連続する低エネルギー区間**がこの ms 未満のときは先頭クリック除去を行わない（語頭の「さ」等が先頭から始まり、UI クリックと誤判定されやすい場合の対策）。`null` 省略時は従来どおりピーク対ノイズ比のみで判定。
   * **末尾**: `trail_click_requires_silence_ms`（任意）は **バースト直前の連続無音**が十分なときだけ末尾除去する（語尾の短い高エネルギー帯を誤切りしない）。
4. **ログ／メタデータ**
   * 除去した先頭・末尾の **サンプル数または ms** を `metadata` に記録し、再現性とチューニングに使う。

#### パイプライン上の位置

* **リサンプル後**（必要なら **ハイパスで DC・ローを除いた後**）に置くと検出が安定しやすい。
* **ノーマライズの前後**は実装で試す。ゲイン前の方が相対閾値が素直な場合と、正規化後の方が閾値共通化しやすい場合がある。**同一設定で再現可能**にすることが優先。
* `trim_silence` とは順序を固定する（推奨: **`edge_ui_click` → `trim_silence`**。先に無音だけ削るとクリックと発話の境界が曖昧になることがあるため）。

### 5.2 MFA による足切り（任意）

* 設定: **`mfa_gate`**（[開発環境.md](./開発環境.md) に **MFA の導入**を記載）。
* **入力波形**: `run_steps_on_array` 済みの float 波形を一時 WAV に書く（**`two_pass_denoise` が無効**なら **単一 `audio_pipeline` 完了後**。**`two_pass_denoise` が有効**で **align/enhance 未指定**なら **第 1 パス＝denoise を除いた `audio_pipeline` 直後**。**完全分離**なら **第 1 パス＝`audio_pipeline_align` 直後**）。対応する **`.lab`** に `text_norm`（`lab_strip_spaces: true` なら空白除去）を1行で格納し、`mfa align` を **バッチ**（実効 `batch_size_resolved`、**§4.1**）で実行する。
* **辞書・音響モデル**: 例として日本語は事前学習の `japanese_mfa`（`mfa model download dictionary japanese_mfa` / `acoustic japanese_mfa`）。パスまたは MFA が解決するモデル名を `dictionary` / `acoustic_model` に指定。
* **MFA 前足切り（任意）**: **`mfa_gate.prefilter`** — `enabled: true` のとき、**`mfa align` を呼ぶ前**に、MFA 入力波形へ **§6 と同じ `run_quality_gate`** をかける。閾値はルートの **`quality_gate`** に **`prefilter.quality_gate_overrides`** をマージしたもの（空なら `quality_gate` と同一）。落ちたクリップは **`mfa_prefilter_<理由>`**（例: `mfa_prefilter_gate_snr`）。**`two_pass_denoise` 有効時は denoise 前の波形**に対するため、厳しすぎる場合は `overrides` で SNR・無音などを緩めるか `prefilter` をオフにする。
* **拒否理由**: `mfa_align_failed`（プロセス失敗・TextGrid 欠落など）、`mfa_phoneme_mismatch`（`compare_phones_to_g2p: true` かつ G2P と **比較用文字列**の編集距離率が `max_token_error_rate_vs_g2p` を超える）。
* **G2P との比較（`compare_phones_to_g2p: true`）**: TextGrid の phones を空白区切りトークン列にしたあと、**`mfa_gate.mfa_to_g2p_token_map_path`**（YAML 辞書）が指定されていれば **先に OpenJTalk G2P 側のトークン列へ変換**してから `phoneme_compare` する（形式は [音素照合マニフェスト.md](./音素照合マニフェスト.md) の MFA トークンマップと同じ）。未指定のときは従来どおり MFA トークン列をそのまま比較する（記号体系が異なると誤拒否しやすい）。
* **注意**: MFA の音素記号と OpenJTalk G2P は **既定で同一ではない**。音素一致で足切りする場合は **`mfa_to_g2p_token_map_path` を推奨**。アライメント成否だけで足切りする場合は **`compare_phones_to_g2p: false`** でよい。
* `report.json` に `mfa_gate` の要約（バッチ回数・設定・マップパス・マップエントリ数）を記録する。

### 5.3 NeMo Forced Aligner（NFA）による足切り（任意）

**MFA（§5.2）とは別系統**の強制アライメント足切り。設定は **`nfa_gate`**（`mfa_gate` と **同時に `enabled: true` にできない**）。NeMo の CTC（または hybrid の **CTC デコード**）上で Viterbi 照合し、**トークン CTM**（`output_dir/ctm/tokens/{utt_id}.ctm`）から比較列を得る。実装は **`NFA_PYTHON`**（未設定時は `python3`）が **`NFA_ALIGN_DIR/align.py`** を **カレント `NFA_ALIGN_DIR` で**起動する。Dev Container では `NFA_PYTHON`・`NFA_ALIGN_DIR` がコンテナ環境に設定される。

* **拒否理由（NFA 専用・MFA 理由コードとは別名）**
  * `nfa_align_failed` — NFA プロセス失敗、マニフェスト／CTM 欠落、モデル読み込み失敗など。
  * `nfa_token_mismatch` —（`compare_tokens_to_g2p: true` 時）CTM のモデル出力トークン列と G2P 参照列の編集距離率が閾値超過。
  * `nfa_pred_text_missing` —（`compare_pred_text_to_norm: true` 時）NeMo 出力に `pred_text` が無い・空。
  * `nfa_ref_phonemes_missing` —（`compare_pred_text_to_norm: true` 時）参照の **`phonemes`**（`text_norm` の OpenJTalk G2P 結果）が空。通常は **`text.phonemize: true`** で前段が失敗していない限り発生しない。
  * `nfa_pred_phonemize_failed` —（`compare_pred_text_to_norm: true` 時）`pred_text` を正規化したうえで **OpenJTalk G2P** に失敗した（不正文字列など）。
  * `nfa_pred_phoneme_mismatch` —（`compare_pred_text_to_norm: true` 時）参照 **`phonemes`** と **`pred_text` 由来の G2P 音素列**の **トークン編集距離率**が `max_pred_phoneme_error_rate_vs_norm` を超える。
  * `nfa_prefilter_<理由>` — NFA バッチ投入**前**の品質ゲート拒否（§5.2 の `mfa_prefilter_*` と**対称だが接頭辞は `nfa_`**。例: `nfa_prefilter_gate_snr`、`nfa_prefilter_gate_duration`、`nfa_prefilter_mora_estimate_failed`）。
* **最終品質ゲート（enhance 後・accept 直前）**: **`nfa_gate.enabled: true`** のとき、§6 の **`run_quality_gate`** にはルート **`quality_gate`** に **`nfa_gate.quality_gate_profile`**（任意・`quality_gate_profiles` から）と **`nfa_gate.quality_gate_overrides`** をマージした閾値を使う（実装: `effective_final_quality_gate`）。Sidon 等で推定 SNR が下がりやすい場合は `min_estimated_snr_db: null` などを `overrides` に書く（`config/default.yaml` のコメント例）。
* **入力波形・テキスト**: MFA（§5.2）と同じタイミングの波形に対し、**`nfa_gate.model_sample_rate_hz`（既定 16000）** へリサンプルした **一時 WAV** と NeMo マニフェスト（JSONL）の **`text`**（`manifest_strip_spaces: true` なら空白除去）でアライメントする。**`nfa_gate.persistent_worker`（既定 ``true``）** のときは **`nfa_align_worker.py` を 1 subprocess 常駐**させモデルを初回のみロードし、各バッチは JSON 1 行でワーカーに渡す。``false`` または環境変数 **`CV_PREPROCESS_NFA_SUBPROCESS=1`** のときは従来どおりバッチごとに **`align.py`** を起動する。いずれも NeMo 側の **`batch_size`** は `nfa_gate.batch_size` を反映する。常駐ワーカー init では **`nfa_gate.worker_chunk_batch_size`**（既定 **32**）を NeMo の `chunk_batch_size` に渡す（バッファードストリーミング利用時や共有メモリが逼迫する場合は **`batch_size`** とあわせて下げる）。**`two_pass_denoise` 有効時**、NFA（子プロセス）と親の SGMSE は **同一クリップを同時に forward することはない**が、**常駐ワーカーは GPU にモデルを載せたまま**なので、親が SGMSE を走らせる間も **VRAM は両方ぶん重なり得る**。ピークを抑えたい場合は **`nfa_gate.release_persistent_worker_before_two_pass_enhance: true`**（第2パス直前に子を終了し次 NFA で再ロード）または **`persistent_worker: false`**（バッチごとにプロセス終了）を検討する。
* **認識文による照合（音素マップ不要）**: **`nfa_gate.align_using_pred_text: true`**（NeMo 既定の「認識してから強制アライメント」）かつ **`nfa_gate.compare_pred_text_to_norm: true`** のとき、NeMo の **`pred_text`** を **`normalize_for_tts`** したあと **OpenJTalk G2P** し、参照の **`phonemes`**（`text_norm` 由来・前段と同じ `g2p_kana`）と **空白区切り音素トークン列の編集距離率**（`max_pred_phoneme_error_rate_vs_norm`）で比較する。**`text.phonemize: true` が必須**（参照音素列が無いと照合できない）。**`compare_tokens_to_g2p` と同時に true にできない**（どちらか一方）。
* **G2P 比較（`compare_tokens_to_g2p: true`）**: **`nfa_gate.nfa_to_g2p_token_map_path`** に YAML（形式は `mfa_to_g2p_token_map_path` と同じだがキーは NFA トークン）を指定し、変換後に `phoneme_compare` する（**`compare_tokens_to_g2p: true` のときは本キー必須**。草案は **`cv-preprocess suggest-nfa-g2p-map`**、[音素照合マニフェスト.md §3.2.2](./音素照合マニフェスト.md)）。マップに無い NFA トークンは従来どおり素通しになる。
* **推奨モデル例**: **`nvidia/parakeet-tdt_ctc-0.6b-ja`** は **`EncDecHybridRNNTCTCBPEModel`** で、NFA が **CTC デコード**に切り替えて扱う。`pretrained_name` と **`model_path`（ローカル .nemo）** は **どちらか一方のみ**（`enabled: true` 時に検証）。
* `report.json` に `nfa_gate` の要約を記録する。

## 6. 受入基準（前処理後の品質ゲート）

「生波形が荒いから除外」ではなく、**前処理後**に次のような基準で採否する（閾値は `config`）。

* 発話時間が `min_duration_sec`〜`max_duration_sec` の範囲内
* **残存**無音比率・リード/トレールが上限以下（無音カット後でも長すぎる無音を弾く）
* クリッピング率、直流オフセット異常
* 推定 SNR やノイズフロア（実装可能な範囲）
* テキスト長と音声長の整合（著しく不自然なもの）

テキスト側の除外（日本語以外、記号過多など）は **前処理の前後どちらで行うか**を設定で選べるようにしてよいが、**音声の採否は原則「4. 全体フロー」の手順 5（受入基準）以降**とする。

## 7. 正規化仕様（テキスト）

* 全角・半角の統一
* 数字の読み方（展開ルールの固定）
* 記号の削減または統一、句読点の統一
* 英字の扱い（読みルールまたは除去ポリシーの固定）
* 空白の正規化、連続記号の圧縮

**実装メモ**: `text_norm` の区切り（単語間スペースの有無・形態素境界かなど）は **学習・推論で同一ルール**になるよう、仕様または設定で固定し、**13 生成レコード例**の形式と一致させる。

## 8. 音素化仕様

* 辞書またはルールベースで **学習時と推論時で同一変換**
* 促音・長音・撥音などのルールを固定
* アクセント情報を使う場合はオプション化し、メタデータに明示

**強制アライメントについて**: 本パイプラインには **MFA / CTC 等のアライメント実行を組み込まない**（別ツール・別ジョブで行う）。

**音素一致による足切り（任意）**: 別プロセスで **音声に対する音素列（判定音素）** を用意し、**JSON Lines マニフェスト**で `validated.tsv` の `path` と対応づけて渡すと、**G2P で得たコーパス側音素列**と **空白区切りトークン列**として比較し、編集距離ベースでずれが大きいクリップを拒否できる（`text.phoneme_alignment_check`、**§8.1**）。**両方の音素表記は同一ルール（記号・区切り）に揃える**こと。`text.phonemize: true` が前提。

### 8.1 `phoneme_alignment_check`（設定）

* `enabled`: 有効化
* `manifest_path`: JSONL。各行 `{"source_path": "<TSV の path と同一>", "phonemes": "<空白区切り音素>"}`
* `missing_manifest_entry`: `skip`（マニフェストに無いクリップは比較しない）／`reject`（無いクリップは拒否）
* `max_token_error_rate`: `0` でトークン列**完全一致**のみ通過。`0〜1` で許容する正規化編集距離（÷ 長い方のトークン数）の上限

`report.json` にマニフェスト件数・設定の要約を記録する。拒否理由コード: `phoneme_alignment_mismatch` / `phoneme_alignment_missing_manifest`。

**マニフェストの作り方**（形式・MFA との記号差・パス注意など）は **[音素照合マニフェスト.md](./音素照合マニフェスト.md)** に分離して記載する。

**実装メモ**: 音素化を行わない場合は `phonemes` を省略するか `null` とするかをスキーマで決め、**13** の例と整合させる。

## 10. train / val / test 分割

公式の `train/dev/test` は用途が異なるため、**本パイプラインで生成した「採用済み発話」集合**を母集団として分割する。

* 比率の例: train 90% / val 5% / test 5%（設定で変更可能）。**3 つの比率の和が 1.0 であること**を設定検証で保証する。
* **同一 `client_id` が train と test（および val）に跨がない**よう **speaker-aware split** を推奨
* **乱数シードを固定**し、同じ入力・設定・バージョンで **分割結果が再現**するようにする（実装では `seed` を `report.json` に記録）。
* 変数名が `dev` のフレームワーク向けには、**val を `dev` として書き出すオプション**（ファイル名・JSON 列の値の両方）を付けてもよい

**実装メモ**: 話者数が極端に少ない場合、speaker-aware では val/test に十分な発話が回らない可能性がある。そのときのフォールバック（警告のみ／エラー／層化の緩和）を実装またはドキュメントで定めるとよい。

## 11. CLI 設計（想定サブコマンド）

* `scan` — 入力コーパス根の走査、`client_id` 一覧や列検証
* `preprocess` — 話者指定・音声チェーンの一括実行
* `phoneme-manifest` — **`phoneme_alignment_check` 用 JSONL** を生成（**OpenJTalk G2P 互換**。`g2p_text` / `mfa_textgrid`＋トークンマップ）。詳細は [音素照合マニフェスト.md](./音素照合マニフェスト.md)
* `secondary` — 一次 `preprocess` 出力に対する二次補正・再品質ゲート（設定 `secondary` が必要。[追加仕様.md §11](./追加仕様.md)）
* `text-normalize`（または `norm-text`）— **テキスト正規化のみ**（**5** の `normalize`（音声）と名前が衝突しないようにする）
* `phonemize` — 音素化
* `gate` / `score` — 受入基準・品質指標の計算（`preprocess` に内包してもよい）
* `split` — train/val/test
* `export` — 学習用出力
* `report` — 集計レポート

```yaml
  train: 0.9
  val: 0.05
  test: 0.05
  seed: 42
```

## 13. 生成レコード例（`metadata.jsonl` 1 行）

* **1 発話 = 1 行**（JSON Lines）。文字コードは **UTF-8**。
* **`utt_id`**: パイプライン内で一意。生成規則（連番、`source_path` のハッシュ、`sentence_id` 併用など）を実装で固定し、**同一入力の再実行で衝突しない**こと。
* **`speaker_id`**: 本仕様では Common Voice の **`client_id` をそのまま格納**してよい。別名を付ける場合は変換ルールを `report.json` に記載する。
* **`source_path`**: TSV の `path` と同じ（`clips/` 相対のファイル名）。

```json
{
  "utt_id": "cv_ja_000001",
  "audio_path": "wavs/cv_ja_000001.wav",
  "text_raw": "今日はいい天気です。",
  "text_norm": "今日 は いい 天気 です",
  "phonemes": "ky o o | w a | i i | t e n k i | d e s u",
  "speaker_id": "client_123",
  "duration_sec": 3.82,
  "silence_ratio": 0.08,
  "estimated_snr_db": 14.2,
  "quality_score": 0.91,
  "split": "train",
  "source_release": "cv-corpus-25.0-2026-03-09",
  "source_path": "clips/xxxx.mp3",
  "audio_pipeline_id": "default_v1",
  "edge_removed_leading_ms": 18.0,
  "edge_removed_trailing_ms": 12.0,
  "edge_click_confidence": "high"
}
```

### 13.1 rejects・レポート（最低限）

* **`rejects.csv`（または設定パス）**: 採用しなかった行ごとに少なくとも **`source_path` または TSV 行を特定できる情報**、**理由コード**（例: `decode_failed`, `early_gate_duration`, `early_gate_snr`, `two_pass_denoise_failed:*`, `mfa_prefilter_gate_duration`, `mfa_prefilter_gate_snr`, `mfa_prefilter_mora_estimate_failed`, `nfa_prefilter_gate_duration`, `nfa_prefilter_gate_snr`, `nfa_prefilter_mora_estimate_failed`, `gate_duration`, `gate_silence`, `text_locale`, `phoneme_alignment_mismatch`, `phoneme_alignment_missing_manifest`, `mfa_align_failed`, `mfa_phoneme_mismatch`, `nfa_align_failed`, `nfa_token_mismatch`, `nfa_pred_text_missing`, `nfa_ref_phonemes_missing`, `nfa_pred_phonemize_failed`, `nfa_pred_phoneme_mismatch`）、**任意で元の `sentence` の抜粋**。列名は実装で固定し、**16 MVP** 完了時点でドキュメント化する。
* **`report.json`**: 少なくとも **入力 `corpus_root`・リリース識別子・`audio_pipeline_id`・処理件数・reject 内訳・`split.seed`**。**17** の運用要件と重なるが、スキーマのバージョン（例: `report_schema_version`）を付けると後方互換が取りやすい。

## 16. MVP の優先順位

1. `validated.tsv` + `clips/` の読み込みと **`include_client_ids` による話者絞り込み**、および任意で **`clip_metadata_filters`（§2.4）**
2. デコード → リサンプル → **`edge_ui_click`（5.1・パラメータは粗めで可）** → ノーマライズ → 無音トリム の **最小チェーン**
3. 日本語・ロケール判定と文字正規化
4. 受入基準（長さ・残存無音率）
5. speaker-aware による train/val/test
6. JSONL 出力と `rejects.csv`
7. ノイズ抑制・ローパス等の **段階的追加**（config の `steps` でON/OFF）
8. **`edge_ui_click` の閾値・スキャン幅のチューニング**（誤検出時は除去しない安全側を維持）

## 17. リリース差分・運用上の注意

* Scripted Speech は **リリースごとに構造・列が変わり得る**。**バージョン文字列と取得日を `report.json` に必ず記録**する。
* `reported.tsv` は通報された文の情報であり、テキスト側のフィルタに使うかはポリシーで決める。
* ノイズ抑制・深層モデル利用時は **ライセンスと再配布可否**に注意する。

## 18. 付録: CPU 側パフォーマンス（実装メモ）

ニューラル推論や `librosa` STFT 以外で、**サンプル長に比例する Python ホットパス**の削減を優先してきた経緯の整理。変更時は品質・再現性・VRAM を検証すること。

| 領域 | 内容 | 主な参照 |
|------|------|----------|
| フレーム RMS（窓・hop 同期） | ベクトル化した窓スキャン | `cv_preprocess/audio/edge_ui_click.py`（`_frame_rms`） |
| 品質ゲート RMS 列 | 窓集合を揃えたベクトル化 | `cv_preprocess/audio/quality_gate.py`（`_frame_rms_peak_series` 等） |
| SNR 推定 | フレームエネルギーの集約 | `cv_preprocess/audio/snr.py` |
| 末尾無音・エッジ探索 | `numpy` / `scipy.ndimage` を用いた短絡 | `quality_gate.py`, `edge_ui_click.py` |
| True 連続区間フィルタ | `scipy.ndimage.label` + `bincount` | `cv_preprocess/audio/binary_runs.py` |
| リップノイズ修復 | 候補区間まわりのベクトル化 | `cv_preprocess/audio/lip_noise_repair.py` |

パイプライン側では **SGMSE マイクロバッチ**、**二段 denoise**、**prefilter と本番品質ゲートの条件付き再利用**（同一設定かつモーラ前提が一致するときの短絡）などを実装済み（`cv_preprocess/audio/sgmse_dereverb.py`、`cv_preprocess/pipeline/preprocess/two_pass.py`、`cv_preprocess/pipeline/preprocess_efficiency.py`、`cv_preprocess/pipeline/preprocess/session.py`、`tests/test_quality_gate_prefilter_reuse.py` など）。

**未着手の案**（NFA/MFA と SGMSE のオーバーラップ、往復リサンプル削減、指紋計算の軽量化など）は Issue / PR で設計し、**同一設定・同一入力での採否・主要メタの一致**（または差分を `report` で説明できる仕様変更）を受け入れ基準とする。
